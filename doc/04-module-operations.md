# Module Operations

Navigation: [Previous: Git And Submodules](03-git-and-submodules.md) | [README](README.md) | [Next: Packaging Workflow](05-packaging-workflow.md)

This page documents how modules are operated from the `babel-chrome` meta workspace: where module sources live, how production zips are built, how they are installed, and how update checks are resolved.

The technical module contract is documented separately in [Module Integration Contract](08-module-integration-contract.md). That page is the canonical reference for manifest fields, stable routes, viewer dispatch, request headers, settings routes, address badges, open-with endpoints, lifecycle hooks, drag-and-drop hooks, message relay, and internal-page context menu conventions.

## Source Ownership

Modules are independent extension projects. The meta workspace references them under `modules/`, but each module owns its own source tree, dependencies, assets, tests, documentation, and release version. The current maintained modules are PHP projects, and BabelChrome also supports static web modules through the `static-web` runtime, local HTTP process modules through the `process-web` runtime, and non-web process modules through the `process-runtime` runtime.

Current module sources:

- `babelforge.demo-module`;
- `babelforge.json-viewer`;
- `babelforge.laravel-module`;
- `babelforge.markdown-viewer`;
- `babelforge.node-process-web-demo`;
- `babelforge.openapi-viewer`;
- `babelforge.plain-php-module`;
- `babelforge.project-launcher`;
- `babelforge.process-runtime-demo`;
- `babelforge.process-web-demo`.

The browser does not load those development directories directly. A module becomes available to BabelChrome only after a production zip has been installed from `babelchrome://modules`.

## Build Zips

From the meta workspace root, build every module:

```bash
./tools/dev2prod.sh
```

Build one module by directory name:

```bash
./tools/dev2prod.sh markdown-viewer-module
```

Build one module by manifest id:

```bash
./tools/dev2prod.sh babelforge.markdown-viewer
```

The generated archives are written to `zip/`. When a module version changes, the packaging workflow keeps older archives under `zip/old/`.

## Install Or Update

Open:

```text
babelchrome://modules
```

Then use `Install or update module zip` and select one or more generated zip files.

Installing a zip whose module id is already present updates the installed module. The module remains independent from the native app bundle. Modules keep any runtime dependencies they need inside their own package or declare host-provided executable paths through `requiredSettings`. Static web modules usually ship only public files, while process-backed modules may ship their own executable runtime dependencies because the host treats those packages as module-owned files.

## Update Checks

The `babelchrome://modules` page can check installed modules against release metadata. BabelChrome stores:

- an optional update URL pointing either to `modules-release-manifest.json` or to a base URL containing that file;
- an optional local update folder containing module zip archives, used as fallback when no URL is configured or when the URL cannot be read.

Remote update checks use the release manifest generated by `tools/prepare-module-release.sh`.

Local update checks scan the configured folder for zip archives. BabelChrome caches each archive path, size, modification time, module id, and module version. When the archive size or modification time changes, BabelChrome reopens the zip and refreshes the cached metadata.

When newer module versions are found, the update report lets the user select the desired updates and install them as a batch.
